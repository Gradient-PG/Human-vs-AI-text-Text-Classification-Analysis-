  • Results from the preliminary GPT-4 run.
  • Methodological issues surfaced during analysis.

The pipeline is a method-agnostic CV fold loop driven by raid_analysis/experiments/runner.py. Every experiment plugs into the same loop by supplying a selector (how neurons are chosen) and an evaluator (how the chosen set is scored). The primary experiment (sparse_probe) performs selection; dependent experiments (ablation, patching, confound) reuse its per-fold selections without re-doing selection, via source_loader.load_source_selections.

Per fold the runner strictly enforces a three-step train/test separation:

  1. Step 1 — Selection: either selector.select(train_acts, train_labels) or a precomputed SelectionResult is loaded. Selection never sees test data.
  2. Step 2 — Evaluation probe: a fresh L2 logistic regression (train_eval_probe) is fit on the full train fold using all features. The eval probe is the fixed measurement instrument; it is not retrained per evaluator.
  3. Step 3 — Evaluation: evaluator.evaluate(test_acts, test_labels, test_meta, selection, eval_probe). Evaluators see test data and the frozen artifacts from steps 1–2.

2.2 Data & Splits

  • Activations: concatenation of all transformer layers, loaded per-generator from results/activations_raid_<slug>/. Global feature axis has D = n_layers × hidden_dim. For GPT-2 this is 12 × 768 = 9,216.
  • Balanced subsampling: _balanced_subsample in data/loading.py draws max_samples / 2 per class with seed 42. Current run uses max_samples = 2000; target final run: ~5000.
  • Splits: StratifiedKFold with a label × domain stratification key. Each fold has balanced class and domain representation, which is required for same-domain patching pairs and domain-level confound diagnostics.
  • Metadata (SampleMetadata): text_lengths, domain_ids, domain_names. There is no explicit “generator” field — generator identity is implicit in which activation directory is loaded.
  • Splits are persisted as splits.json per experiment and reused across dependent experiments for the same run_id.

2.3 Selection: SparseProbeSelector (L1)

Implementation: raid_analysis/selection/sparse_probe.py.

C sweep and knee detection

  1. Split the outer train fold into inner train (80%) / inner val (20%) via StratifiedShuffleSplit.
  2. For each C ∈ C_values (default [0.001, 0.01, 0.1, 1.0, 10.0]), fit L1 logistic regression (penalty='l1', solver='saga') on inner train; record (C, val_accuracy, n_nonzero).
  3. Pick the knee in the (n_nonzero, val_accuracy) curve using the maximum-perpendicular-distance heuristic: draw a chord from the sparsest to densest point, pick the point farthest from the chord. Returns the C_values index.
  4. Refit L1 on the full outer train fold at that knee C. Report selected neurons and a ranking by absolute coefficient magnitude.

SelectionResult payload (persisted per fold)

  • neuron_indices: set of (layer, neuron) tuples with non-zero coefficients.
  • ranking: (layer, neuron) ordered by descending |coef| (used as top-k order in ablation and patching).
  • probe, scaler: the fitted L1 probe and its StandardScaler (used only for the diagnostic selector_test_accuracy).
  • train_mean: per-dimension training-fold mean (used by mean ablation to avoid test leakage).
  • metadata: C, solver, n_nonzero, n_features, train_accuracy, sweep_curve, knee_idx, knee_C, C_values.

2.4 Evaluation Probe (L2)

Implementation: raid_analysis/evaluation/probe_factory.py.

  • StandardScaler fit on the full train fold activations.
  • LogisticRegression(C=eval_probe_C, solver='lbfgs', ...) — L2 default, all D features.
  • Serialized per fold as l2_probe_weights.npy, l2_probe_intercept.npy, l2_scaler_mean.npy, l2_scaler_scale.npy.
  • The L2 probe is independent of what the selector chose. This is the critical design fact that shapes ablation interpretation (see §4.1).

for (seed, fold) in splits_by_seed:

    train_lbl   = labels[fold.train_idx]

                 # OR: precomputed_selections[(seed, fold_idx)]

    eval_probe = train_eval_probe(train_acts, train_lbl, C=..., ...)  # L2, all features

    metrics = evaluator.evaluate(test_acts, test_lbl, test_meta, selection, eval_probe)

Aggregation across folds is done in _aggregate_metrics: mean/std/min/max over scalar metrics. Nested metrics (e.g. ablation_sweep lists) are not averaged into aggregate.json; they live per-fold in eval_metrics.json.

2.6 Dependent Experiments

Ablation, patching, and confound all set source_experiment: sparse_probe in YAML. The runner:

  1. Resolves source_dir to results/experiments/<run_id>/sparse_probe/<generator>.
  2. Loads SelectionResult per fold via load_source_selections (reads selection.json + train_mean.npy).
  3. Passes these as precomputed_selections; Step 1 is skipped. Step 2 (L2 probe) is still fit fresh from the current config.

Reuse ensures ablation/patching/confound operate on exactly the same neuron sets, rankings, and splits as the primary sparse_probe run.

Ablation evaluator (evaluation/ablation.py)

  • For each k in k_values, take top-k global indices from selection.ranking.
  • Mean-ablate those columns in test activations: replace with selection.train_mean[idx]. Score L2 eval probe.
  • Random-k baseline: draw n_random_seeds random k-subsets (seed 42 — this is deterministic across folds; see §4.3).
  • Report per-k: selected_acc, selected_drop, random_acc_mean/std, random_drop_mean/std.

Patching evaluator (evaluation/patching.py)

  • Restrict to human test samples (label==0) that have at least one same-domain AI sample in the test fold (valid_humans).
  • Compute base_preds = eval_probe.predict(base_acts) once.
  • For each of n_shuffles random donor assignments (permuted within domain), copy donor_activations[:, top_k_indices] into base_acts[:, top_k_indices]. Score eval_probe.predict(patched). A “flip” is human (0) → AI (1).
  • Flip rate per shuffle = flips / |valid_humans|. Mean/std over shuffles.
  • Random-k baseline: n_random_draws random neuron subsets; for each, run n_shuffles pairings and report mean per-draw, then mean/std across draws.
  • Per-domain breakdown at the largest k used.

Confound evaluator (evaluation/confound.py, to be removed)

  • For each selected neuron, on the pooled test fold: Spearman(act, text_length) and one-way ANOVA(act, domain_id).
  • “Flagged” if p < 0.05 under either test. No multiple-comparison correction. Tests are on pooled human+AI samples, which conflates class signal with domain association (see §4.2).

Interpretation

The extended k-sweep (new k = -1 sentinel) reshapes the story. Four things stand out:

  1. Flip rate grows monotonically with k all the way to the full selected set on both generators. No saturation in the selected range — adding every selected neuron keeps adding sufficiency signal. This is the expected signature of a distributed-but-targeted representation.
  2. At k = n_selected, the ratio to random is ~18× (GPT-4) and ~12× (Mistral). Both far outside any plausible noise band. The ratio is the scientifically meaningful signal and is consistent across generators.
  3. Absolute flip rates are now non-trivial. GPT-4: 3.6% at n_selected (vs 1.65% at k=100 in the previous report); Mistral: 25.7% at n_selected. The concern from the previous run (“is the absolute rate just noise?”) is largely retired.
  4. GPT-4 is harder to flip than Mistral. GPT-4’s L2 probe is very confident (AUC 0.998), so a patched sample must traverse a larger margin; Mistral’s probe (AUC 0.970) has more borderline samples that a targeted patch can reliably tip. See §3.6.
Is the absolute rate still a problem? Substantially less so after the full-set sweep. GPT-4 at 3.6% average (vs 0.20% random) and Mistral at 25.7% (vs 2.16% random) are clearly above any plausible noise. The remaining honest caveats:
  • Flip rate is a thresholded metric. Probability-shift ΔP(AI) would be more sensitive and is still planned (§5.1 (G)).
  • Single-seed. Generator-level aggregate is stable (5-fold std is tight: GPT-4 selected flip at n_selected is 0.036 ± 0.005 across folds), but per-domain breakdowns are not (see callout above).
  • No saturation. If the representation were concentrated, flip rate should plateau well before n_selected. Continued growth is itself informative — it confirms the redundancy story from ablation.

Two findings, one mild and one striking:

Per-domain patching breakdowns are not reliable at this sample size (see §3.3 caveat). The numbers here are sufficient to frame hypotheses for the final run; they are not sufficient to close them.

4. Methodological Findings

4.1 Redundancy story (ablation ↔ patching)

The apparent contradiction — “ablation is weak, patching works” — is not a contradiction once one recognizes that:

  1. Ablation tests necessity against an L2 probe that has access to all 9,216 features. In a redundant representation, removing a small fraction is compensated by the remainder.
  2. Patching tests sufficiency by injecting genuine AI-class signal on human rows. Wrong-class signal in selected features competes with human signal elsewhere and occasionally wins; random-feature injections essentially never win.

The extended k-sweep (§3.2, §3.3) makes this quantitative on both generators: ablating the full selected set drops accuracy by 0.05 pp (GPT-4) and 1.35 pp (Mistral), while patching the full set achieves 3.6% and 25.7% flip rates respectively — in both cases 12–18× above random. The two experiments measure opposite properties of the same redundant, distributed representation; both confirm it.

An alternative “restricted eval probe” design (L2 trained only on selected features) was considered and rejected because it trivializes the high-k end of the ablation sweep (you ablate the whole input). It is worth running separately for the smaller-model claim — see §5.1.

4.2 Per-neuron confound test — why we are dropping it

  • Pooled tests conflate class with domain. A genuine AI detector whose activation varies across domains because AI text differs stylistically across domains produces a significant domain ANOVA, even though nothing confounding is happening.
  • No multiple-comparison correction. Fixable but doesn’t rescue the deeper issue.
  • Marginal ≠ confounded classification. Per-neuron marginal tests do not tell you whether the classifier’s decision boundary actually depends on domain. Cross-domain generalization does.
  • 99% flagged is uninformative. The result does not discriminate hypotheses.

The simpler, stronger alternative is a combination of (i) leave-one-domain-out generalization and (ii) a domain probe trained on selected features. Both are in the plan.

4.3 Selection vs evaluation separation (minor notes)

  • The L1 selection probe is persisted but only used for the diagnostic selector_test_accuracy metric. The evaluator always scores a fresh L2 probe. Good separation.
  • Ablation random-k uses np.random.RandomState(42) per call, which is the same seed for every k and every fold. Random-k baselines are therefore highly correlated across the sweep. Not wrong, but worth noting; future versions should derive the random seed from fold/seed/k.
  • Patching similarly uses a fixed seed for shuffles and neuron draws. Same caveat.
  • LogisticRegression with penalty='l1', solver='saga' — verified correct against sklearn docs (initial concern retracted).

5. Plan Going Forward

5.1 New experiments

(A) Cross-domain generalization primary

  • Leave-one-domain-out: train on 5 domains, test on the 6th. Repeat 6 times.
  • Full pipeline per fold: run L1 selection and fit L2 eval probe on the 5 training domains, evaluate on the held-out domain.
  • Report: per-domain held-out accuracy, average across held-out domains, overlap of selected neurons across the 6 runs (additional stability signal).
  • This directly tests the paper’s interpretability claim (“neurons encode AI-vs-human signal, not domain”) and its smaller-model-transfer claim.
  • Implementation cost: a new splitter in data/splits.py (~50 LoC); no runner/evaluator changes.

(B) Restricted probe / “smaller model” test primary

  • Take the stable neuron set (or per-fold selected set).
  • Train an L2 probe only on those features on the train fold; evaluate on the test fold.
  • Report: accuracy with ~167 features vs baseline (~9,216 features). Also cross-domain version (combines with (A)).
  • Implementation cost: new evaluator variant or a flag on ProbeAccuracyEvaluator to accept a feature-restriction.

(C) Domain probe baseline diagnostic

  • Train an L2 probe on activations to predict domain (6-class). Two variants: all 9,216 features vs only selected neurons.
  • If selected neurons predict domain at near chance (16.7%) / only modestly above, they don’t encode domain appreciably. If they predict it near-perfectly, there’s a legitimate confound concern.
  • Cheap: one extra evaluator, runs once per fold.

(D) Multi-seed validation mandatory

  • Current seeds: [42]. Target: at least [42, 123, 456].
  • Stability threshold unchanged at 0.8 (≥ 12/15 across 3 seeds × 5 folds).
  • Multiplies runtime by seed count.

(E) Characterize multi-generator

  • Already in the catalog but not run. Compares stable neuron sets across generators.
  • Executed after primary runs on headline generators are done.

(F) Sample size sensitivity supplementary

  • Standalone, outside main pipeline. Re-run sparse_probe at n ∈ {1000, 2000, 5000, all} to check stability of findings.
  • Single generator, single seed sufficient.

(G) Patching / ablation — extended k sweep + probability shift partially done

  • done Extended k_values with a -1 sentinel interpreted as n_selected in both AblationEvaluator and PatchingEvaluator (see raid_analysis/evaluation/ablation.py and patching.py). Applied in runs 20260418_011033 and 20260418_011044. Outcome summarised in §3.2 / §3.3: ablation has a clear ceiling (negligible drop even at full set); patching shows monotone growth with no saturation up to n_selected.
  • still planned Add small supersets beyond n_selected (e.g. 200, 300) to probe whether flip rate keeps climbing past the L1 knee or plateaus — cheap diagnostic for how exhaustive the selected set really is.
  • still planned Add a probability-shift metric alongside flip rate: mean ΔP(AI) = eval_probe.predict_proba(patched)[:,1] - eval_probe.predict_proba(base)[:,1], reported per k, per domain, with random-k baseline. This is continuous and complementary to the thresholded flip metric (see §3.3 callout). Purely an evaluator upgrade; no selection changes required.

5.2 Pipeline changes

  • Remove confound evaluator, its config, and factory registration.
  • Add leave-one-domain-out splitter to data/splits.py.
  • Add restricted-feature evaluator or flag.
  • Add domain-prediction evaluator.
  • done Dynamic k (= per-fold n_selected) supported via k = -1 sentinel in both AblationEvaluator and PatchingEvaluator. still planned Emit a probability-shift metric in addition to flip rate.
  • Fix random seeding in ablation/patching baselines to depend on fold/seed/k (low priority; purely hygiene).

5.3 Tiered generator strategy

Given 11 generators and ~3 days of final compute, not all generators receive the same analysis.

TierGeneratorsExperiments
Headline (2–3) Tentative: one from each family (GPT-4, a Mistral, a LLaMA). Final selection TBD, criteria: architecture-family spread first, size/generation second. Full suite: sparse_probe, ablation, patching, cross-domain, restricted probe, domain probe, multi-seed.
Reduced (rest) Remaining 8–9 generators. sparse_probe only (standard 5-fold, maybe 1 seed).

Paper placement: headline generators in main text, reduced-suite generators in an appendix (“we verified the approach across N generators”). Decided at writing time; for now the structural choice is what matters.

5.4 Compute budget plan

Current run wall-clock (2k samples, 1 seed, 5 folds, GPT-4, C grid of 5): ~2h for the full pipeline. Target final run changes that parameter vector substantially:

  • n_samples: 2000 → ~5000 (~2.5×)
  • C values: 5 → more (TBD; likely 1.5–2× in the sweep step only)
  • seeds: 1 → 3 (3×)
  • cross-domain folds: 5 random → 6 leave-one-out on top of k-fold (additional run; roughly 1× per headline generator)
  • generators: 1 → 2–3 headline (full) + 8–9 reduced (sparse_probe only)
Action before committing Pick one generator (GPT-4), run one full pipeline cycle at target settings (5k samples, expanded C grid, 1 seed, including cross-domain) on the best available GPU. Measure wall-clock. Multiply: 
total_hours ≈ single_cycle_hours
             × seed_count
             × headline_generator_count
             + sparse_probe_hours × reduced_generator_count
If this exceeds ~60h (2.5 days, with buffer), reduce to 2 headline generators or drop a seed. Do this measurement before launching the final run.
